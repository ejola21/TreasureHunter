# Image Target AR 보물찾기 서비스 — 완벽 가이드
## Marker-based AR for Treasure Hunt (vs GPS-based)

> 본 문서는 `sechon_play_00.md` 의 보완 명세로, GPS 기반이 아닌 **이미지 마커 기반 AR** 로 보물찾기 서비스를 구축하는 방안을 다룬다.
> PlaySpot 의 본래 정체성인 "Treasure Hunter" 를 시각적으로 구현하는 핵심 기술 선택지.

> **결론 한 줄**: 충분히 가능, 오히려 GPS 기반보다 "보물찾기다움" 이 강함. 단 마커 운영 (인쇄·설치·유지보수) 이라는 새로운 운영 비용 발생.

---

## 1. Image Target 이 보물찾기에 강한 이유

### GPS 보물찾기의 한계

| 한계 | 설명 |
|---|---|
| 5~10m 오차 | 도심 빌딩 그늘에서 "여기인 것 같은데?" 회색 영역 |
| **실내 0** | 박물관·시장 안에서 작동 안 함 |
| **시각 임팩트 0** | 핀에 도달했다는 알림만 — 게임감 약함 |
| 평면 메카닉 | 도착 = 발견. "찾는 행위" 의 재미 부족 |
| 위치 위조 가능 | Mock Location 으로 어디서든 트리거 |

### Image Target 의 본질적 강점

| 강점 | 의미 |
|---|---|
| **실제로 봐야 인식** | 사용자가 진짜 그 자리에 가서 카메라를 들고 사물을 찾아야 함 → "탐험감" |
| **cm 단위 정확도** | 마커 인식 시점에 정확한 3D 위치 → 그 위에 AR 보물 떠오름 |
| **실내·지하·터널 모두 OK** | GPS 의존 0 |
| **AR 보상의 시각 임팩트** | 보물상자가 화면에 튀어나오는 카타르시스 |
| **위조 불가** | 마커 사진을 다른 곳에서 보여줘도 시각·각도가 어색해서 시스템이 알 수 있음 (조명·왜곡 검증) |

→ **GPS 가 "어디로 가라" 면, Image Target 은 "거기서 무엇을 찾아라"**. 후자가 진짜 보물찾기.

## 2. 동작 원리 — 처음부터 끝까지 이해하기

### 2-1. 핵심 오해 풀기

> ❌ **마커 = QR 코드 같은 특수 패턴이 있어야 한다**
> ✅ **마커 = 그냥 사진. 시각적으로 충분히 복잡하면 어떤 사진이든 마커가 됨**

ARKit/ARCore 입장에서 "마커" 는 **사전에 등록한 어떤 평면 이미지** 다. QR 도, 흰 정사각형 안의 패턴도 필요 없다. **간판 사진 한 장** 이면 충분.

### 2-2. 비유로 이해 — 사람이 친구 얼굴 알아보는 과정

```
1. 사전 학습: 친구 얼굴을 평소에 봐서 기억해둠
   ↓
2. 거리에서: 군중 속 얼굴 하나하나를 봄
   ↓
3. 매칭: "어, 저 사람 내 친구잖아!" 인식
```

Image Target 도 똑같다:

```
1. 사전 등록: 김밥집 간판 사진을 시스템에 미리 학습시킴
   ↓
2. 카메라 작동: 시장에서 실시간으로 영상 캡쳐
   ↓
3. 매칭: 화면 안에 등록된 간판이 보이는 순간 즉시 인식
```

### 2-3. "사전 학습" 의 실체

ARKit/ARCore 가 사진을 받으면 내부적으로:

```
사진 1장 (예: 김밥집 간판)
     ↓
[Step 1] sRGB 변환
     ↓
[Step 2] 특징점 추출 (코너·에지·점)
       — 사진 위의 200~2000개 "고유한 점"을 좌표로 저장
     ↓
[Step 3] 특징 디스크립터 생성
       — 각 점 주변의 픽셀 패턴을 64~256byte 숫자열로 변환
     ↓
[Step 4] 데이터베이스 저장 (.arobject / .imgdb)
       — 마커 1장 = 약 10~50KB
```

시각화:

```
원본 사진:               특징점 추출 (시스템 내부):
┌──────────┐              ┌──────────┐
│  김밥집   │              │ . . ..   │
│  맛있어요 │   →          │  .::. .. │  ← 점·코너 위치만 기억
│   ★ ★ ★  │              │ .* .* .. │
└──────────┘              └──────────┘
```

이 "특징점 지도" 만 저장하지, 원본 사진은 폰에 안 둔다.

### 2-4. "실시간 인식" 의 실체

```
카메라가 1초에 30번 캡쳐 (30 FPS)
     ↓
매 프레임마다:
  [a] 현재 화면에서도 특징점 추출
  [b] 등록된 마커들의 특징점과 비교
  [c] 매칭 점이 50개 이상이면 → "인식 성공"
  [d] 매칭된 점들의 위치 차이로 3D 각도·거리 계산
     ↓
"김밥집이 카메라 정면 1.5m 거리, 약간 우측 15° 기울어짐" 정보 반환
     ↓
이 정보로 AR 보물상자를 정확한 위치에 합성
```

→ 사람 눈으로 보는 것 같은 메커니즘. 다만 0.05초 만에.

### 2-5. 무엇을 마커로 삼을 수 있나 — 체크리스트

**✅ 좋은 마커 (인식률 90%+)**

| 종류 | 예 | 이유 |
|---|---|---|
| 손글씨 간판 | "통인김밥" 붓글씨 | 모서리·곡선 풍부 |
| 인쇄 포스터 | 영화 포스터, 메뉴판 | 색·텍스트 명확 |
| 무늬 있는 그림 | 민화·동양화 | 비대칭·복잡 |
| 음각·양각 명패 | 동판 안내판 | 그림자가 특징 |
| 잡지 표지·책 표지 | 책방의 책등 | 텍스트+그래픽 |
| 모자이크 타일 | 카페 벽장식 | 점·선 풍부 |
| 직접 인쇄한 스티커 | 디자인된 보물 마커 | 완벽한 통제 가능 |

**⚠️ 나쁜 마커 (인식률 30~60%)**

| 종류 | 예 | 이유 |
|---|---|---|
| 단색 간판 | "Open" 흰 바탕 검은 글자만 | 특징점 부족 |
| 그라데이션 벽 | 카페 벽지 | 매끈해서 점이 없음 |
| 반사 표면 | 거울·유리·금속 | 매번 다르게 보임 |
| 디지털 화면 | 가게 LCD 사이니지 | 콘텐츠 바뀜 |
| 빛나는 네온사인 | 야간 네온 | 글로우로 형태 흐려짐 |
| 잎·식물 | 화분, 화단 | 바람에 움직임 |
| 거리가 먼 큰 건물 | 경복궁 정면 전체 | 사용자 거리에 따라 크기 변함 |

**❌ 마커로 쓸 수 없는 것**

| 종류 | 이유 |
|---|---|
| 사람 얼굴 | 표정·각도 변화 |
| 빈 종이/벽 | 특징점 0 |
| 거의 같은 무늬 | 시장 내 같은 가게 간판 다수 → 어떤 게 어떤 건지 구별 불가 |
| 3D 입체 (조각·동상) | Image Target 은 평면만. (3D 는 별도 ARKit Object Anchor) |

### 2-6. "고정물" 의 4가지 의미 차원

**"간판이나 고정물 같이 변하지 않는 거가 있으면 되나?"** — 네, 정확하다. 단 "변하지 않는" 의 4가지 차원을 구분해야 함:

| 변화 차원 | 영향 | 예 |
|---|---|---|
| **물리적 위치** | 결정적 | 간판이 떼이거나 옮겨지면 ✗ |
| **시각적 외관** | 결정적 | 간판이 새로 칠해지면 ✗ |
| **조명** | 일시적 | 햇빛 방향 바뀌면 일시 인식률↓, 다시 OK |
| **부분 가림** | 일시적 | 앞에 사람 지나가도 OK |

→ 즉, **물리적+시각적으로 안정적** 인 것이 좋은 후보.

서촌 적용 예시:

| 후보 | 평가 |
|---|---|
| 대오서점 60년 된 간판 | ⭐⭐⭐⭐⭐ 안정·텍스처 풍부 |
| 박노수미술관 안내판 | ⭐⭐⭐⭐⭐ 박물관 안내판은 거의 안 바뀜 |
| 통인시장 김밥집 간판 | ⭐⭐⭐⭐ 가게 폐업 리스크 5% |
| 시장 내 메뉴판 (인쇄) | ⭐⭐⭐ 분기마다 바뀔 수 있음 |
| 카페 LED 사인보드 | ⭐ 화면 바뀜 |
| 가로수 | ✗ 잎·계절 변화 |
| 사람 동상 | ⭐ 평면 인식만 가능, 정면에서만 |

### 2-7. 사전 준비 (Pre-work) 단계별 체크리스트

**Phase 1. 후보 선정 (디자이너·기획자 작업)**
```
□ 코스 안에서 마커 후보 10~20곳 리스트업
□ 각 후보의 "안정성" 평가 (위 표 기준)
□ 후보 상점·기관과 사전 협의 (시각 사용 허가)
□ 정확한 GPS 위치 기록
□ 마커 주변 환경 메모 (조명·각도)
```

**Phase 2. 마커 촬영 (현장 작업)**
```
□ 카메라 정면, 약 1m 거리에서 촬영
□ 사진 정면 + 좌·우 30° 추가 (백업)
□ 균일한 조명 (그늘 아래, 흐린 날 권장)
□ 손떨림 없는 선명한 사진
□ 해상도 1024 × 1024 이상
□ 마커 실측 (가로 또는 세로 cm)
□ 사진에 다른 사물 안 들어가게 (간판만)
□ 같은 후보를 시간대·날씨 다르게 3장씩 (Phase 3 에서 비교)
```

**Phase 3. 시스템 등록 (백오피스 작업)**
```
□ 디자이너 도구에서 마커 사진 업로드
□ 실측 크기 입력 (예: width=60cm)
□ 라벨 입력 ("Daeo Bookstore Sign")
□ AR 콘텐츠 선택 (보물상자 / 캐릭터 / 글자)
□ 단서 텍스트 입력 ("60년 된 책방 간판")
□ 보상 설정 (포인트·뱃지)
□ 시스템이 자동으로:
  - sRGB 변환
  - 특징점 추출
  - 품질 점수 표시 (★★★★★)
  - 품질 낮으면 "재촬영 권장" 경고
```

**Phase 4. 컴파일·배포**
```
□ 모든 마커를 .arobject (iOS) / .imgdb (Android) 파일로 컴파일
□ 서버에 업로드
□ 앱이 백그라운드로 다운로드 (사용자 모름)
□ 앱이 카메라 켜는 순간 자동 로드
```

**Phase 5. 출시 후 유지보수**
```
□ 사용자 인식 시도·성공 로깅
□ 인식률 70% 이하 마커 → 자동 알림
□ 분기별 현장 점검
□ 손상·교체·폐업 시 즉시 비활성화
```

### 2-9. 사용자 입장 경험 — 재정리

```
1. 앱 열고 "통인시장 도시락 수사대" 시작
       ↓
2. 단서: "빨간 떡볶이 옆 노란 김밥집 간판"
       ↓
3. 사용자: 시장 안 돌아다님 (앱은 카메라 OFF — 배터리 절약)
       ↓
4. 사용자: 의심되는 간판 보임 → "Scan" 버튼 탭
       ↓
5. 카메라 ON, AR 세션 시작 (마커 DB 자동 로드, 0.5초)
       ↓
6. 사용자: 폰을 간판 쪽으로 향함
       ↓
7. 시스템: 0.05초 만에 매칭 → "Detected!"
       ↓
8. AR 보물상자 ✨ 간판 위로 떠오르며 열림 (사운드 + 햅틱)
       ↓
9. 보상 획득, 다음 단서 노출
       ↓
10. 카메라 OFF, 사용자 다음 마커 찾으러 이동
```

핵심: **사용자가 폰을 들이대는 시점에만 카메라 활성**. 항상 켜져 있지 않음.

### 2-10. 자주 묻는 질문 (FAQ)

**Q1. 같은 김밥집 간판이 시장 안에 2개 있으면?**
→ 시스템이 둘 다 동일 마커로 인식 → 사용자 혼란.
**해결**: 사전 검수 단계에서 중복 확인 + 두 간판의 미세 차이를 기준으로 구별 (예: 그늘 위치) 또는 둘 다 등록해서 둘 다 정답으로.

**Q2. 마커 사진을 인터넷에서 가져와도 되나?**
→ **저작권 주의**. 가게 간판은 가게 주인의 자산. 협의 후 사용 권장. 무허가 사용 시 분쟁 위험.

**Q3. 사용자가 마커 사진을 찍어서 보여줘도 인식되나?**
→ 네, 인식된다. 이게 **위조 가능성**. 방지하려면:
- 마커 등록 시 "AR 보물은 실제 마커 앞에서만 트리거" 옵션
- 시스템이 카메라의 미세한 움직임·각도 변화를 검사 → 정지 화면은 거부
- (완벽한 방지는 어렵지만 99% 케이스 차단)

**Q4. 비 오는 날도 작동?**
→ 카메라 렌즈에 물방울 → 인식률 30~50% ↓. 사용자에게 "맑은 날 권장" 안내.

**Q5. 어두운 곳도?**
→ 폰 플래시 자동 활성화 (옵션). 다만 야간 미션은 별도 시나리오로.

**Q6. 마커 1장당 데이터 크기?**
→ 10~50KB. 100개 등록해도 5MB 이하. 앱 용량 부담 없음.

**Q7. 인터넷 없이도 작동?**
→ 마커 DB 가 일단 다운로드되면 오프라인 작동 가능. 단, 보상 동기화는 다시 온라인 시.

**Q8. 카메라 권한 거부하면?**
→ 마커 인식 불가능. 폴백: 사용자가 단서 텍스트만 보고 정답 입력 (예: 가게 이름).

### 2-11. 한 줄 정리 (이 섹션 요약)

> **"마커" = 사전에 등록한 시각적으로 풍부한 평면 이미지.**
> **준비 = 사진 1장 + 실측 크기 + AR 콘텐츠 + 시스템 등록.**
> **간판·안내판·포스터·고유한 그림 모두 마커가 될 수 있음.**
> **단 평면·풍부한 특징·물리적으로 안정해야 함.**

### 2-12. 실제 PoC 1시간 시도 — 추천 흐름

체감 1시간이 자료 100페이지보다 정확하다. 본격 진행 전 빠른 검증:

1. **이번 주말 1시간 시도** — 폰 + Reality Composer (iOS 무료) 또는 ARCore Augmented Images 샘플 앱
2. **노트북 옆 책 한 권을 마커로 등록** → 폰으로 책을 비추면 책 위에 3D 큐브 뜨는 거 확인
3. **체감 확인** → 진짜 보물찾기로 쓸 만한지 본인이 느껴봄
4. **본격 진행 결정**

PoC 가이드가 필요하면 단계별로 추가 안내 가능.

## 3. 무엇이 가능한가 — 8가지 메카닉

### 메카닉 1. 단일 마커 → AR 보물 발현
```
사용자: 단서 "통인시장 김밥집 간판" 받음
       → 시장 안 돌아다니며 김밥집 발견
       → 폰 들이댐
       → 간판이 인식되는 순간 AR 보물상자가
         간판 위로 떠오르고 열림
       → 코인·뱃지·쿠폰 획득
```

**핵심 재미**: 발견의 순간 = AR 애니메이션. 0.5초의 시각적 만족.

### 메카닉 2. 연쇄 마커 (스토리 시퀀스)
```
마커 1 → 단서: "다음은 빨간 우체통 옆 가게"
마커 2 → 단서: "그 가게 안쪽 그림에 답이 있다"
마커 3 → 그림 인식 → 보물상자
```

5개 마커 = 30~60분 분량 미니 게임. 모든 마커 클리어 시 "스토리 컬렉션" 완성.

### 메카닉 3. 다중 마커 동시 인식
```
박노수미술관 정원의 4개 석등 모두 한 화면에 잡으면
→ 4개가 빛나며 연결됨
→ 가상 정원 무늬 떠오름
```

여러 마커가 보이는 풍경에서 **한 번에 모아야** 클리어. 사용자가 자리를 잡는 행위 자체가 게임.

### 메카닉 4. 마커 + 사용자 행동 조합
```
마커 인식 후:
- 폰을 돌려서 AR 화살표 방향 따라가기
- 화면 위 AR 버튼 탭
- 마이크에 단어 외치기
```

마커가 미니게임의 진입점. 인식 후엔 다른 메카닉이 따라옴.

### 메카닉 5. 숨겨진 마커
```
마커가 일반인이 보면 "그냥 무늬" 처럼 보이도록 디자인
(예: 한옥 문살 패턴, 추상 디자인)
사용자만 단서 (예: "처마 끝 격자 무늬") 로 인식
```

지나가는 사람이 무심코 지나치는 사물이 보물의 단서가 됨. **"내가 발견한" 비밀** 의 매력.

### 메카닉 6. AR Cloud Anchor — 사용자 간 공유
```
사용자 A 가 골목에 AR 보물 숨김 (Cloud Anchor 등록)
→ 다른 사용자 B 가 같은 자리에서 찾음
```

UGC 보물찾기. 친구가 만든 보물을 다른 친구가 찾는 SNS 적 확장.

### 메카닉 7. AR 풍경 합성
```
경복궁 광화문 인식 → 200년 전 광화문 (조선시대) AR 합성
사용자가 폰을 통해 보면 옛 모습 + 인물·말·갓 등장
```

Then vs Now 의 AR 강화 버전. 단순 사진 비교가 아닌 **현장 위에 시간이 입혀짐**.

### 메카닉 8. 인쇄 마커 + 디지털 미션
```
스티커·엽서·티켓에 마커 인쇄
→ 카페·게스트하우스에서 배포
→ 스캔 시 보너스 미션 발화
```

오프라인 ↔ 온라인 연결. 이벤트·콜라보 활용 강력.

## 7. 마커 디자인·운영 노하우

### 7-1. 좋은 마커의 5가지 조건

| 조건 | 설명 | 예 |
|---|---|---|
| 풍부한 텍스처 | 모서리·점·곡선이 많음 | 손글씨 간판 ⭐, 단색 벽 ✗ |
| 비대칭 | 회전 식별 가능 | 정사각 로고 ✗, 비대칭 무늬 ⭐ |
| 고대비 | 흑백 변환해도 구별 | 그레이톤 ✗ |
| 평면 | 종이·금속판·필름 | 굴곡진 천 ✗ |
| 고유성 | 비슷한 사물 X | 시장에 같은 간판 10개 ✗ |

### 7-2. 마커 종류별 추천 매체

| 종류 | 비용/장 | 수명 | 적합 |
|---|---|---|---|
| 일반 종이 인쇄 | ₩100 | 1~2주 | 단기 이벤트 |
| 코팅 종이 | ₩500 | 1~3개월 | 시즌 캠페인 |
| **PVC 스티커** | ₩2,000 | 6개월~2년 | 일반 보물찾기 ⭐ |
| 메탈 플레이트 | ₩30,000 | 5~10년 | 영구 설치 |
| 인테리어 데칼 | ₩10,000 | 2~5년 | 상점 협업 |
| 기존 간판 활용 | ₩0 | 기존 수명 | 가장 저렴 ⭐ |

### 7-3. 마커 운영 라이프사이클

```
1. 사진 촬영 (정면·고품질·균일 조명)
   ↓
2. ARKit/ARCore DB 컴파일 (.arobject / .imgdb)
   ↓
3. 서버 업로드 + 메타 (실측 크기, 위치, 단서)
   ↓
4. 클라이언트 동기화 (앱이 백그라운드로 다운로드)
   ↓
5. 인식 실패율 모니터링
   - 자동 (사용자 인식 시도 vs 성공 로깅)
   - 수동 (사용자 신고 버튼)
   ↓
6. 신고 5건 누적 → 운영자 알림
   ↓
7. 현장 점검 → 재인쇄 / 재촬영 / 비활성화
```

### 7-4. 실외 마커 관리 비용 (현실)

| 항목 | 연간 비용 (마커 10개 기준) |
|---|---|
| 초기 제작·설치 (PVC) | ₩500,000 |
| 분기 점검 (1인 4시간 × 4회) | ₩400,000 |
| 손상 교체 (연 30%) | ₩600,000 |
| **합계** | **₩1,500,000 / 년** |

**기존 간판 활용 모드** 면 거의 0 — 하지만 가게 폐업·간판 교체 리스크는 존재.

## 8. 한계와 함정

### 8-1. 인식 실패 원인 (실제 출시 후 데이터)

| 원인 | 비율 | 대응 |
|---|---|---|
| 어두운 조명 (저녁·실내) | 25% | 플래시 자동 활성화 |
| 역광 (햇빛 정면) | 20% | "각도를 바꿔보세요" 가이드 |
| 마커 손상 (낙서·기름) | 15% | 모니터링 + 교체 |
| 카메라 너무 가까움 (< 30cm) | 10% | 가이드 박스 표시 |
| 너무 멈 (> 2m) | 15% | "더 가까이" 안내 |
| 마커 가려짐 (사람·물건) | 10% | 사용자가 직관적 해결 |
| 인식 알고리즘 한계 | 5% | 기술적 한계 |

### 8-2. 사용자가 좌절하는 순간 (방지 필수)

1. **3번 스캔 실패 → 다른 마커 같은 사진?** — 자동 힌트 활성
2. **인식 성공 후 AR 안 뜸** — 로딩 0.5초 이내 보장 필수
3. **걸어가다 보니 마커 없는 자리** — 단서가 너무 어려움. 점진적 힌트 제공
4. **단서 따라가도 안 찾음** — 지도 보조 옵션 (선택)

### 8-3. 기술적 함정

- **Flutter AR 플러그인 불안정** — 네이티브 채널 직접 작성 필수
- **iOS 12 미만 / Android 8 미만 미지원** — 사용자 5~10% 폴백 필요
- **HDR 사진 마커는 인식 안 됨** — sRGB 변환 필수
- **AR 세션 메모리 누수** — 5분 이상 연속 세션은 강제 재시작
- **iCloud 동기화 충돌** — 마커 DB 가 큰 경우

### 8-4. 운영적 함정

- **상점 협의 시간 과소평가** — 가게마다 1~2시간 면담 필요
- **마커 위치 사진을 외부에 공개하면 안 됨** — 사람들이 직접 찾는 재미가 사라짐
- **계절 변화 영향** — 잎이 마커 가림, 눈이 덮음
- **건물 리모델링** — 마커 자리가 사라짐. 사전 계약 명시 필요

## 9. GPS vs Image Target — 보물찾기 관점 비교

| 항목 | GPS 보물찾기 | Image Target 보물찾기 |
|---|---|---|
| 정확도 | 5~10m | <1cm (인식 후) |
| 실내 | ✗ | ✓ |
| 시각 임팩트 | 약 | **강 (AR 효과)** |
| 발견의 카타르시스 | "도착 알림" | **"보물 발현 애니메이션"** |
| 운영 비용 | 0 | 마커 관리 |
| 위조 가능성 | Mock GPS 로 가능 | 매우 어려움 |
| 초기 출시 속도 | 빠름 | 3개월 |
| 사용자 디바이스 호환 | 99% | 70~80% (AR 지원 기기) |
| 단서 풀이 재미 | 약 | **강** |
| 친구·가족 동반 적합 | 중 | **상 (모여서 본다)** |
| 마케팅 임팩트 | 중 | **상 (SNS 영상)** |
| 비 오는 날 | OK | 카메라 가림 ✗ |
| 밤 | OK | 조명 필요 ✗ |

**결론**: 두 방식은 보완재. **GPS = 광역 동선, Image Target = 정밀 발견**.

## 10. 권장 보물찾기 출시 전략

### Step 1 — GPS 기반 (기존 PlaySpot) 유지
스팟 도착·동선 안내는 GPS 그대로.

### Step 2 — 핵심 스팟 5~10곳에 Image Target 추가
각 스팟에서 "도착 후 마커 찾아 스캔" 단계 신설 → 보물 발견의 카타르시스 강화.

### Step 3 — 신규 "보물찾기 모드" 출시
GPS 무관 100% Image Target 기반의 새 게임 모드:
- 통인시장 도시락 수사대 (시장 내부)
- 박노수미술관 보물찾기 (실내)
- 야간 도깨비 사냥 (이벤트성)

### Step 4 — 상점·이벤트 협업
인쇄 마커 배포 → 카페 쿠폰·콜라보 미션

## 12. 솔직한 평가

### 기술적 가능성: ⭐⭐⭐⭐⭐
ARKit/ARCore Image Target 은 2018년부터 성숙. 안정성·정확도 검증 완료.

### Flutter 통합 난이도: ⭐⭐⭐⭐
네이티브 채널 필수. 직접 Swift/Kotlin 작성 필요. PoC 1주 권장.

### 보물찾기 적합성: ⭐⭐⭐⭐⭐
GPS 기반보다 본질적으로 더 "보물찾기다움". 발견의 카타르시스 강력.

### 운영 부담: ⭐⭐⭐
마커 인쇄·설치·점검 인력 필요. 기존 간판 활용 시 부담 ↓.

### 사용자 임팩트: ⭐⭐⭐⭐⭐
SNS 공유율 GPS 미션 대비 2~3배 (AR 영상이 시각적으로 강함).

### 비용: ⭐⭐⭐⭐
초기 개발 3개월, 운영 연 ₩150만 (마커 10개), 클라우드 비용 거의 0.

## 14. 한 줄 결론

> **"PlaySpot = 보물찾기" 가 되려면 GPS 가 아니라 Image Target 이 필요합니다.**
>
> GPS 만으로는 "어디에 갔는가" 만 추적. Image Target 으로는 "거기서 무엇을 발견했는가" 가 가능.
> Phase 1 (PoC 6주) 으로 시작해서 "도착 후 마커 스캔 → AR 보물 발현" 한 가지 메카닉만 검증해도 사용자 반응이 완전히 달라집니다.
>
> 기존 PlaySpot 의 정체성 (Treasure Hunter) 을 마침내 시각적으로 구현하는 핵심 기술입니다.
